# Лабораторна 13 — Захист індивідуального проекту

## Мета

Презентувати та захистити свій індивідуальний проект з тестування. Продемонструвати здатність застосовувати техніки тестування, вивчені протягом курсу, до реального застосунку.

## Вимоги до проекту

### Тестований застосунок

Оберіть або створіть невеликий ASP.NET Core Web API застосунок, що включає:

- Щонайменше **3 API-ендпоінти** (CRUD-операції)
- **Рівень бази даних** з використанням Entity Framework Core (будь-який провайдер)
- Щонайменше **1 залежність від зовнішнього сервісу** (наприклад, email, платежі, сповіщення)
- **Валідацію вхідних даних** щонайменше на 2 ендпоінтах
- **Бізнес-логіку** з нетривіальними правилами (не просто прохідний CRUD)

Ви можете використати проект з попереднього курсу, особистий проект або створити новий. Приклади предметних областей:

| Предметна область | Ендпоінти | Бізнес-логіка |
|---|---|---|
| Менеджер завдань | CRUD завдань, призначення користувача, зміна статусу | Переходи між статусами, валідація дати виконання |
| Електронна комерція | товари, замовлення, кошик | Перевірка наявності, правила знижок, підсумки замовлень |
| Бібліотечна система | книги, читачі, видачі | Ліміти видачі, штрафи за прострочення, наявність |
| Система бронювання | кімнати, бронювання, користувачі | Конфлікти дат, перевірка місткості, політика скасування |

### Необхідні типи тестів

Ваш проект повинен включати всі наступні:

| Тип тестів | Мінімальна кількість | Інструменти |
|---|---|---|
| **Модульні тести** | 15 | xUnit v3, Shouldly, NSubstitute |
| **Інтеграційні тести** (API) | 8 | WebApplicationFactory |
| **Тести бази даних** | 6 | Testcontainers (SQL Server або PostgreSQL) |
| **Тести продуктивності** | 3 сценарії | k6 |

**Загальний мінімум: 32 тести** (модульні + інтеграційні + бази даних) + 3 сценарії k6

### Очікування щодо якості тестів

- Тести дотримуються **шаблону AAA** (Arrange-Act-Assert)
- Назви тестів використовують конвенцію `MethodName_Scenario_ExpectedBehavior`
- **Без залежностей між тестами** — кожен тест виконується незалежно
- Мокування використовується доречно (зовнішні сервіси, а не внутрішня логіка)
- Покриття коду **≥ 80%** (вимірюється за допомогою Coverlet)

### Сценарії продуктивності k6

Включіть щонайменше 3 скрипти k6:

1. **Smoke-тест** — 1-2 VU, 30 секунд, перевірка базової лінії
2. **Навантажувальний тест** — 10-50 VU, 1-2 хвилини, перевірка продуктивності під нормальним навантаженням
3. **Стрес-тест** — поступове збільшення до точки відмови

Кожен скрипт повинен включати:
- `checks` для перевірки статусу відповіді та вмісту
- `thresholds` для часу відповіді (наприклад, p95 < 500ms)

## Процедура захисту

### Тривалість: 15-20 хвилин на студента

### Частина 1 — Презентація (5-7 хв)

1. Коротко опишіть предметну область та архітектуру вашого застосунку
2. Покажіть структуру проекту та організацію тестів
3. Представте звіт покриття коду
4. Покажіть результати тестів продуктивності k6

### Частина 2 — Жива демонстрація (5-7 хв)

1. Запустіть усі тести: `dotnet test --verbosity normal`
2. Запустіть один сценарій k6 наживо
3. Покажіть звіт покриття: `dotnet test --collect:"XPlat Code Coverage"`

### Частина 3 — Запитання та відповіді (5 хв)

Очікуйте запитання на кшталт:
- "Чому ви обрали мокування X замість використання реальної реалізації?"
- "Що станеться, якщо видалити цей тест — який баг може проникнути?"
- "Як би ви протестували цей граничний випадок?"
- "Що означає цей поріг k6 і чому ви встановили саме таке значення?"
- "Чому це інтеграційний тест, а не модульний?"

## Оцінювання

| Критерії | Вага |
|---|---|
| Модульні тести (якість, покриття, використання NSubstitute) | 20% |
| Інтеграційні тести (WebApplicationFactory, HTTP-перевірки) | 20% |
| Тести бази даних (Testcontainers, реальна поведінка БД) | 20% |
| Тести продуктивності k6 (сценарії, перевірки, пороги) | 15% |
| Покриття коду ≥ 80% | 10% |
| Якість презентації та захисту | 15% |

## Здача роботи

- Завантажте ваш проект до **репозиторію GitHub**
- Включіть `README.md` з:
  - Описом проекту
  - Як зібрати та запустити (`dotnet build`, `dotnet run`)
  - Як запустити тести (`dotnet test`)
  - Як запустити тести k6
  - Як згенерувати звіт покриття
- Надішліть URL репозиторію до сесії захисту
