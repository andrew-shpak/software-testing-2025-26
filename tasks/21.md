# Task 21: Hospital Patient Management

## Domain

Hospital system for managing patients, doctors, and medical records.

## Entities

- **Patient**: Id, FirstName, LastName, DateOfBirth, Gender, BloodType, Phone, EmergencyContact
- **Doctor**: Id, FirstName, LastName, Specialization, LicenseNumber, IsAvailable
- **MedicalRecord**: Id, PatientId, DoctorId, Diagnosis, Treatment, Notes, VisitDate, FollowUpDate

## Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| GET | /api/patients | Get all patients |
| POST | /api/patients | Register a patient |
| GET | /api/patients/{id} | Get patient with medical history |
| GET | /api/doctors | Get doctors (filter by specialization, availability) |
| POST | /api/doctors | Register a doctor |
| POST | /api/records | Create a medical record |
| GET | /api/patients/{id}/records | Get medical records for a patient |
| GET | /api/doctors/{id}/patients | Get patients treated by a doctor |

## Business Rules

- DateOfBirth cannot be in the future
- Doctor LicenseNumber must be unique
- Medical record must reference existing patient and doctor
- BloodType must be valid (A+, A-, B+, B-, AB+, AB-, O+, O-)
- FollowUpDate must be after VisitDate

## Testing Scope

- **Unit Tests**: Blood type validation, date validations, follow-up scheduling logic
- **Integration Tests (WebApplicationFactory)**: Patient registration, record creation, medical history retrieval
- **Database Tests (Testcontainers)**: Patient-doctor-record relationships, license uniqueness, chronological record queries
- **Performance Tests (k6)**: Load test patient search, stress test medical record creation

## AutoFixture Fields

Fields to auto-generate with AutoFixture (not business-critical, just need valid data):

- **Patient**: FirstName, LastName, Phone, EmergencyContact
- **Doctor**: FirstName, LastName, Specialization
- **MedicalRecord**: Diagnosis, Treatment, Notes

> Fields like DateOfBirth, Gender, BloodType, LicenseNumber, IsAvailable, VisitDate, FollowUpDate, PatientId, DoctorId have business rules and should be set explicitly in tests.

## Database

Use **PostgreSQL** as the database engine. For Testcontainers, use `Testcontainers.PostgreSql` package.

## Database Seeding

For performance and integration tests, the database must be pre-seeded with **at least 10,000 records** distributed across all entities. Use AutoFixture (or Bogus) to generate realistic test data. Example distribution should reflect real-world ratios between parent and child entities.
