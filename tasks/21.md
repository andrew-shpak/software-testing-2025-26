# Завдання 21: Управління пацієнтами лікарні

## Домен

Лікарняна система для управління пацієнтами, лікарями та медичними записами.

## Сутності

- **Patient**: Id, FirstName, LastName, DateOfBirth, Gender, BloodType, Phone, EmergencyContact
- **Doctor**: Id, FirstName, LastName, Specialization, LicenseNumber, IsAvailable
- **MedicalRecord**: Id, PatientId, DoctorId, Diagnosis, Treatment, Notes, VisitDate, FollowUpDate

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/patients | Отримати всіх пацієнтів |
| POST | /api/patients | Зареєструвати пацієнта |
| GET | /api/patients/{id} | Отримати пацієнта з медичною історією |
| GET | /api/doctors | Отримати лікарів (фільтр за спеціалізацією, доступністю) |
| POST | /api/doctors | Зареєструвати лікаря |
| POST | /api/records | Створити медичний запис |
| GET | /api/patients/{id}/records | Отримати медичні записи пацієнта |
| GET | /api/doctors/{id}/patients | Отримати пацієнтів, яких лікував лікар |

## Бізнес-правила

- DateOfBirth не може бути в майбутньому
- LicenseNumber лікаря має бути унікальним
- Медичний запис повинен посилатися на існуючого пацієнта та лікаря
- BloodType має бути валідним (A+, A-, B+, B-, AB+, AB-, O+, O-)
- FollowUpDate має бути після VisitDate

## Обсяг тестування

- **Юніт-тести**: Валідація групи крові, валідація дат, логіка планування повторного візиту
- **Інтеграційні тести (WebApplicationFactory)**: Реєстрація пацієнта, створення запису, отримання медичної історії
- **Тести бази даних (Testcontainers)**: Зв'язки пацієнт-лікар-запис, унікальність ліцензії, хронологічні запити записів
- **Тести продуктивності (k6)**: Навантажувальне тестування пошуку пацієнтів, стрес-тестування створення медичних записів

## Поля AutoFixture

Поля для автогенерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Patient**: FirstName, LastName, Phone, EmergencyContact
- **Doctor**: FirstName, LastName, Specialization
- **MedicalRecord**: Diagnosis, Treatment, Notes

> Поля DateOfBirth, Gender, BloodType, LicenseNumber, IsAvailable, VisitDate, FollowUpDate, PatientId, DoctorId мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як СУБД. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних повинна бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Розподіл має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
