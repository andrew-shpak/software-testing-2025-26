# Завдання 10: Блог-платформа

## Предметна область

Платформа для ведення блогів з публікаціями, коментарями та тегами.

## Сутності

- **Post**: Id, Title, Content, AuthorName, Slug, PublishedAt, IsPublished, ViewCount
- **Comment**: Id, PostId, AuthorName, Content, CreatedAt, IsApproved
- **Tag**: Id, Name
- **PostTag**: PostId, TagId

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/posts | Отримати опубліковані пости (з пагінацією, фільтр за тегом) |
| GET | /api/posts/{slug} | Отримати пост за slug (збільшує лічильник переглядів) |
| POST | /api/posts | Створити чернетку поста |
| PUT | /api/posts/{id} | Оновити пост |
| PATCH | /api/posts/{id}/publish | Опублікувати чернетку поста |
| DELETE | /api/posts/{id} | Видалити пост |
| POST | /api/posts/{id}/comments | Додати коментар |
| GET | /api/posts/{id}/comments | Отримати схвалені коментарі до поста |
| PATCH | /api/comments/{id}/approve | Схвалити коментар |

## Бізнес-правила

- Slug має автоматично генеруватися з заголовка та бути унікальним
- У списку видно лише опубліковані пости
- Лічильник переглядів збільшується при кожному GET за slug
- Коментарі потребують схвалення перед відображенням
- Не можна опублікувати пост з порожнім вмістом

## Обсяг тестування

- **Модульні тести**: Генерація slug, логіка лічильника переглядів, процес схвалення коментарів
- **Інтеграційні тести (WebApplicationFactory)**: Процес від чернетки до публікації, модерація коментарів, отримання за slug
- **Тести бази даних (Testcontainers)**: Обмеження унікальності slug, конкурентність лічильника переглядів, зв'язок багато-до-багатьох для тегів
- **Тести продуктивності (k6)**: Навантажувальний тест списку постів з пагінацією, стрес-тест конкурентного збільшення лічильника переглядів

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Post**: Title, Content, AuthorName, PublishedAt
- **Comment**: AuthorName, Content, CreatedAt
- **Tag**: Name

> Поля Slug, IsPublished, ViewCount, IsApproved, PostId мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних має бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Розподіл має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
