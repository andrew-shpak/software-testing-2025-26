# Task 38: Courier Service

## Domain

Package delivery and courier tracking system.

## Entities

- **Package**: Id, SenderId, RecipientName, RecipientAddress, RecipientPhone, Weight, Dimensions, Status (Created/PickedUp/InTransit/OutForDelivery/Delivered/Returned), TrackingCode, CreatedAt
- **Sender**: Id, Name, Email, Phone, Address
- **DeliveryUpdate**: Id, PackageId, Status, Location, Notes, Timestamp, UpdatedBy

## Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| POST | /api/packages | Create a package for delivery |
| GET | /api/packages/{trackingCode} | Track package by tracking code |
| GET | /api/packages/{trackingCode}/updates | Get delivery updates timeline |
| POST | /api/packages/{id}/update | Add a delivery status update |
| PATCH | /api/packages/{id}/deliver | Mark as delivered |
| PATCH | /api/packages/{id}/return | Mark as returned |
| GET | /api/senders/{id}/packages | Get sender's packages |
| GET | /api/packages?status={status} | Get packages by status |

## Business Rules

- TrackingCode auto-generated (unique, alphanumeric, 12 chars)
- Status must follow sequence: Created → PickedUp → InTransit → OutForDelivery → Delivered
- Weight must be positive, max 50kg
- Cannot update status of delivered/returned packages
- Each status change creates a DeliveryUpdate record

## Testing Scope

- **Unit Tests**: Tracking code generation, status sequence validation, weight limits
- **Integration Tests (WebApplicationFactory)**: Package lifecycle, tracking endpoint, delivery updates timeline
- **Database Tests (Testcontainers)**: Tracking code uniqueness, status update audit trail, sender-package relationship
- **Performance Tests (k6)**: Load test package tracking, stress test concurrent status updates

## Database

Use **PostgreSQL** as the database engine. For Testcontainers, use `Testcontainers.PostgreSql` package.

## Database Seeding

For performance and integration tests, the database must be pre-seeded with **at least 10,000 records** distributed across all entities. Use AutoFixture (or Bogus) to generate realistic test data. Example distribution should reflect real-world ratios between parent and child entities.
