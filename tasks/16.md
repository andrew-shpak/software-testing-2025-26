# Завдання 16: Продаж квитків на події

## Домен

Платформа управління подіями та продажу квитків.

## Сутності

- **Event**: Id, Title, Description, Venue, Date, StartTime, EndTime, TotalTickets, AvailableTickets, TicketPrice
- **Ticket**: Id, EventId, BuyerName, BuyerEmail, PurchaseDate, TicketCode, IsUsed
- **Venue**: Id, Name, Address, Capacity

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/events | Отримати майбутні події (фільтр за датою, місцем проведення) |
| POST | /api/events | Створити подію |
| GET | /api/events/{id} | Отримати деталі події з доступністю |
| PUT | /api/events/{id} | Оновити подію |
| POST | /api/events/{id}/tickets | Придбати квиток(и) |
| GET | /api/tickets/{code} | Перевірити квиток за кодом |
| PATCH | /api/tickets/{code}/use | Позначити квиток як використаний (сканування при вході) |
| GET | /api/events/{id}/attendees | Отримати список власників квитків |

## Бізнес-правила

- Не можна придбати більше квитків, ніж доступно
- TicketCode має бути унікальним (автоматично згенерований UUID)
- TotalTickets події не може перевищувати місткість місця проведення
- Не можна придбати квитки на минулі події
- Квиток може бути використаний лише один раз

## Обсяг тестування

- **Юніт-тести**: Генерація коду квитка, перевірка доступності, валідація минулих подій
- **Інтеграційні тести (WebApplicationFactory)**: Процес покупки, валідація квитка, запобігання повторному використанню
- **Тести бази даних (Testcontainers)**: Атомарність зменшення доступних квитків, унікальність коду квитка, обмеження місткості місця
- **Тести продуктивності (k6)**: Навантажувальне тестування списку подій, стрес-тестування одночасної покупки квитків (сценарій швидкого розпродажу)

## Поля AutoFixture

Поля для автогенерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Event**: Title, Description
- **Ticket**: BuyerName, BuyerEmail, PurchaseDate
- **Venue**: Name, Address

> Поля Date, StartTime, EndTime, TotalTickets, AvailableTickets, TicketPrice, TicketCode, IsUsed, Capacity мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як СУБД. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних повинна бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Розподіл має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
