# Завдання 11: Кошик електронної комерції

## Домен

Система кошика покупок з товарами, управлінням кошиком та оформленням замовлень.

## Сутності

- **Product**: Id, Name, Description, Price, StockQuantity, Category, SKU
- **Cart**: Id, UserId, CreatedAt, UpdatedAt
- **CartItem**: Id, CartId, ProductId, Quantity, UnitPrice
- **Order**: Id, UserId, Items (знімок), TotalAmount, Status (Pending/Paid/Shipped), CreatedAt

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/products | Отримати товари (фільтр за категорією, діапазоном цін) |
| POST | /api/products | Створити товар |
| GET | /api/cart | Отримати поточний кошик користувача |
| POST | /api/cart/items | Додати товар до кошика |
| PUT | /api/cart/items/{productId} | Оновити кількість товару |
| DELETE | /api/cart/items/{productId} | Видалити товар з кошика |
| POST | /api/cart/checkout | Оформити замовлення з кошика |
| GET | /api/orders | Отримати замовлення користувача |

## Бізнес-правила

- Не можна додати більше товарів, ніж є на складі
- Оформлення замовлення зменшує залишок на складі та очищає кошик
- SKU має бути унікальним
- Ціна при оформленні замовлення фіксується (не залежить від подальших змін ціни)
- Порожній кошик не може бути оформлений

## Обсяг тестування

- **Юніт-тести**: Валідація залишків, розрахунок загальної суми, логіка фіксації ціни
- **Інтеграційні тести (WebApplicationFactory)**: Процес додавання до кошика, процес оформлення замовлення, помилки валідації залишків
- **Тести бази даних (Testcontainers)**: Атомарність зменшення залишків, цілісність знімка замовлення, унікальність SKU
- **Тести продуктивності (k6)**: Навантажувальне тестування списку товарів, стрес-тестування одночасного оформлення замовлень (стан гонки за залишки)

## Поля AutoFixture

Поля для автогенерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Product**: Name, Description, Category
- **Cart**: CreatedAt, UpdatedAt
- **Order**: CreatedAt

> Поля Price, StockQuantity, SKU, Quantity, UnitPrice, TotalAmount, Status, UserId мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як СУБД. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних повинна бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Розподіл має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
