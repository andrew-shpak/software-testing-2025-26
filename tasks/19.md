# Task 19: Job Board

## Domain

Job listing platform with companies, job postings, and applications.

## Entities

- **Company**: Id, Name, Description, Website, Industry, LogoUrl
- **JobPosting**: Id, CompanyId, Title, Description, Location, SalaryMin, SalaryMax, Type (FullTime/PartTime/Contract), PostedAt, ExpiresAt, IsActive
- **Application**: Id, JobPostingId, ApplicantName, Email, ResumeUrl, CoverLetter, AppliedAt, Status (Pending/Reviewed/Accepted/Rejected)

## Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| GET | /api/jobs | Get active jobs (filter by location, type, salary range) |
| POST | /api/jobs | Create a job posting |
| GET | /api/jobs/{id} | Get job details |
| PUT | /api/jobs/{id} | Update a job posting |
| DELETE | /api/jobs/{id} | Close a job posting |
| POST | /api/jobs/{id}/apply | Submit an application |
| GET | /api/jobs/{id}/applications | Get applications for a job |
| PATCH | /api/applications/{id}/status | Update application status |

## Business Rules

- SalaryMin must be less than SalaryMax
- Expired jobs are automatically inactive
- Cannot apply to inactive/expired jobs
- One application per email per job posting
- ExpiresAt must be in the future when creating

## Testing Scope

- **Unit Tests**: Salary range validation, expiration logic, duplicate application check
- **Integration Tests (WebApplicationFactory)**: Job posting flow, application submission, status updates, expired job filtering
- **Database Tests (Testcontainers)**: Unique application constraint, expiration-based queries, company-job cascade
- **Performance Tests (k6)**: Load test job search with multiple filters, stress test application submission

## AutoFixture Fields

Fields to auto-generate with AutoFixture (not business-critical, just need valid data):

- **Company**: Name, Description, Website, Industry, LogoUrl
- **JobPosting**: Title, Description, Location, PostedAt
- **Application**: ApplicantName, ResumeUrl, CoverLetter, AppliedAt

> Fields like SalaryMin, SalaryMax, Type, ExpiresAt, IsActive, Email, Status have business rules and should be set explicitly in tests.

## Database

Use **PostgreSQL** as the database engine. For Testcontainers, use `Testcontainers.PostgreSql` package.

## Database Seeding

For performance and integration tests, the database must be pre-seeded with **at least 10,000 records** distributed across all entities. Use AutoFixture (or Bogus) to generate realistic test data. Example distribution should reflect real-world ratios between parent and child entities.
