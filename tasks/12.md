# Завдання 12: Система інвентаризації

## Домен

Управління складськими запасами з товарами, рухом запасів та постачальниками.

## Сутності

- **Product**: Id, Name, SKU, Category, MinStockLevel, CurrentStock, Unit
- **StockMovement**: Id, ProductId, Type (In/Out/Adjustment), Quantity, Reason, Timestamp, PerformedBy
- **Supplier**: Id, Name, ContactEmail, Phone

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/products | Отримати всі товари (фільтр за категорією) |
| POST | /api/products | Створити товар |
| PUT | /api/products/{id} | Оновити інформацію про товар |
| POST | /api/products/{id}/stock-in | Зафіксувати надходження на склад |
| POST | /api/products/{id}/stock-out | Зафіксувати відвантаження зі складу |
| GET | /api/products/{id}/movements | Отримати історію руху запасів |
| GET | /api/products/low-stock | Отримати товари з рівнем нижче мінімального |
| GET | /api/suppliers | Отримати всіх постачальників |
| POST | /api/suppliers | Додати постачальника |

## Бізнес-правила

- Відвантаження не може перевищувати поточний запас
- CurrentStock оновлюється автоматично при кожному русі
- SKU має бути унікальним
- MinStockLevel має бути невід'ємним
- Попередження про низький запас, коли CurrentStock < MinStockLevel

## Обсяг тестування

- **Юніт-тести**: Розрахунок запасів, виявлення низького рівня запасів, валідація руху
- **Інтеграційні тести (WebApplicationFactory)**: Процес надходження/відвантаження, ендпоінт низького запасу, історія руху
- **Тести бази даних (Testcontainers)**: Узгодженість запасів після руху, запобігання одночасному відвантаженню, аудит-трейл руху
- **Тести продуктивності (k6)**: Навантажувальне тестування списку товарів, стрес-тестування швидких рухів запасів

## Поля AutoFixture

Поля для автогенерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Product**: Name, Category, Unit
- **StockMovement**: Reason, Timestamp, PerformedBy
- **Supplier**: Name, ContactEmail, Phone

> Поля SKU, MinStockLevel, CurrentStock, Type (In/Out), Quantity, ProductId мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як СУБД. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних повинна бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Розподіл має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
