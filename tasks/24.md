# Завдання 24: Трекер витрат

## Предметна область

Персональний фінансовий трекер для витрат, бюджетів та звітів.

## Сутності

- **Expense**: Id, UserId, Amount, Description, Category, Date, PaymentMethod (Cash/Card/Transfer)
- **Budget**: Id, UserId, Category, MonthlyLimit, Month, Year
- **Category**: Id, Name, Icon, Color

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/expenses | Отримати витрати (фільтр за категорією, діапазоном дат, методом оплати) |
| POST | /api/expenses | Додати витрату |
| PUT | /api/expenses/{id} | Оновити витрату |
| DELETE | /api/expenses/{id} | Видалити витрату |
| GET | /api/budgets | Отримати бюджети за поточний місяць |
| POST | /api/budgets | Встановити місячний бюджет |
| PUT | /api/budgets/{id} | Оновити бюджет |
| GET | /api/reports/monthly?month={m}&year={y} | Отримати місячний звіт витрат за категоріями |
| GET | /api/reports/summary | Отримати підсумок витрат (загальна сума, середнє, топ-категорія) |

## Бізнес-правила

- Amount має бути додатним
- Не можна встановити два бюджети для однієї категорії в одному місяці
- Попередження про бюджет, коли витрати досягають 80% ліміту
- Прапорець перевищення бюджету, коли витрати > ліміту
- Дата не може бути в майбутньому

## Обсяг тестування

- **Модульні тести**: Обчислення порогових значень бюджету, агрегація звітів, валідація дати
- **Інтеграційні тести (WebApplicationFactory)**: CRUD витрат, сповіщення про бюджет, генерація місячних звітів
- **Тести бази даних (Testcontainers)**: Унікальність бюджету за категорією/місяцем, запити агрегації витрат, каскадне видалення категорій
- **Тести продуктивності (k6)**: Навантажувальне тестування генерації звітів, стрес-тестування масового створення витрат

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Expense**: Description
- **Budget**: CreatedAt
- **Category**: Name, Icon, Color

> Поля Amount, Category, Date, PaymentMethod, MonthlyLimit, Month, Year, UserId мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних має бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Приклад розподілу має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
