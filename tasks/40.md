# Task 40: Equipment Rental

## Domain

Equipment rental business with inventory, rentals, and maintenance tracking.

## Entities

- **Equipment**: Id, Name, Category (Tools/Electronics/Vehicles/Sports), DailyRate, Status (Available/Rented/Maintenance), Condition (New/Good/Fair/Poor), SerialNumber
- **Rental**: Id, EquipmentId, CustomerName, CustomerEmail, StartDate, EndDate, TotalCost, Status (Active/Returned/Overdue), ReturnCondition
- **MaintenanceLog**: Id, EquipmentId, Type (Routine/Repair), Description, Cost, ScheduledDate, CompletedDate

## Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| GET | /api/equipment | Get equipment (filter by category, status, availability for dates) |
| POST | /api/equipment | Add equipment |
| PUT | /api/equipment/{id} | Update equipment |
| POST | /api/rentals | Create a rental |
| POST | /api/rentals/{id}/return | Return equipment |
| GET | /api/rentals | Get rentals (filter by status) |
| GET | /api/rentals/overdue | Get overdue rentals |
| POST | /api/equipment/{id}/maintenance | Schedule maintenance |
| GET | /api/equipment/{id}/history | Get rental and maintenance history |

## Business Rules

- Cannot rent equipment that is rented or in maintenance
- TotalCost = days × DailyRate (minimum 1 day)
- Overdue if not returned by EndDate
- Overdue surcharge: 1.5× daily rate per extra day
- Equipment goes to Maintenance status after return in Fair/Poor condition
- SerialNumber must be unique

## Testing Scope

- **Unit Tests**: Cost calculation, overdue surcharge, condition-based maintenance trigger
- **Integration Tests (WebApplicationFactory)**: Rental lifecycle, return with condition, overdue detection
- **Database Tests (Testcontainers)**: Serial number uniqueness, availability date range queries, rental history
- **Performance Tests (k6)**: Load test availability search, stress test concurrent rentals

For performance and integration tests, the database must be pre-seeded with **at least 10,000 records** distributed across all entities. Use AutoFixture (or Bogus) to generate realistic test data. Example distribution should reflect real-world ratios between parent and child entities.
