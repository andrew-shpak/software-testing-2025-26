# Завдання 40: Прокат обладнання

## Предметна область

Бізнес з прокату обладнання з інвентарем, орендою та відстеженням технічного обслуговування.

## Сутності

- **Equipment**: Id, Name, Category (Tools/Electronics/Vehicles/Sports), DailyRate, Status (Available/Rented/Maintenance), Condition (New/Good/Fair/Poor), SerialNumber
- **Rental**: Id, EquipmentId, CustomerName, CustomerEmail, StartDate, EndDate, TotalCost, Status (Active/Returned/Overdue), ReturnCondition
- **MaintenanceLog**: Id, EquipmentId, Type (Routine/Repair), Description, Cost, ScheduledDate, CompletedDate

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/equipment | Отримати обладнання (фільтр за категорією, статусом, доступністю на дати) |
| POST | /api/equipment | Додати обладнання |
| PUT | /api/equipment/{id} | Оновити обладнання |
| POST | /api/rentals | Створити оренду |
| POST | /api/rentals/{id}/return | Повернути обладнання |
| GET | /api/rentals | Отримати оренди (фільтр за статусом) |
| GET | /api/rentals/overdue | Отримати прострочені оренди |
| POST | /api/equipment/{id}/maintenance | Запланувати технічне обслуговування |
| GET | /api/equipment/{id}/history | Отримати історію оренд та обслуговування |

## Бізнес-правила

- Не можна орендувати обладнання, яке вже орендоване або на обслуговуванні
- TotalCost = дні x DailyRate (мінімум 1 день)
- Прострочено, якщо не повернено до EndDate
- Надбавка за прострочення: 1.5x денної ставки за кожний додатковий день
- Обладнання переходить у статус Maintenance після повернення у стані Fair/Poor
- SerialNumber повинен бути унікальним

## Обсяг тестування

- **Модульні тести**: Розрахунок вартості, надбавка за прострочення, тригер обслуговування на основі стану
- **Інтеграційні тести (WebApplicationFactory)**: Життєвий цикл оренди, повернення зі станом, виявлення прострочення
- **Тести бази даних (Testcontainers)**: Унікальність серійного номера, запити доступності за діапазоном дат, історія оренд
- **Тести продуктивності (k6)**: Навантажувальне тестування пошуку доступності, стрес-тестування одночасних оренд

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Equipment**: Name
- **Rental**: CustomerName, CustomerEmail
- **MaintenanceLog**: Description

> Поля Category, DailyRate, Status, Condition, SerialNumber, StartDate, EndDate, TotalCost, ReturnCondition, Type, Cost, ScheduledDate, CompletedDate мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних повинна бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Приклад розподілу повинен відображати реальні співвідношення між батьківськими та дочірніми сутностями.
