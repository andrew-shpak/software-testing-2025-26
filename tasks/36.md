# Завдання 36: Система зворотного зв'язку

## Предметна область

Система зворотного зв'язку та запитів на функціональність продукту з голосуванням та відстеженням статусу.

## Сутності

- **Feedback**: Id, Title, Description, Type (Bug/Feature/Improvement), Status (Open/UnderReview/Planned/InProgress/Done/Closed), Priority (Low/Medium/High), AuthorName, AuthorEmail, CreatedAt, VoteCount
- **Vote**: Id, FeedbackId, VoterEmail, CreatedAt
- **Comment**: Id, FeedbackId, AuthorName, Content, CreatedAt, IsOfficial

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/feedback | Отримати весь зворотний зв'язок (фільтр за типом, статусом, сортування за голосами) |
| POST | /api/feedback | Надіслати зворотний зв'язок |
| GET | /api/feedback/{id} | Отримати зворотний зв'язок з коментарями |
| PUT | /api/feedback/{id} | Оновити зворотний зв'язок |
| PATCH | /api/feedback/{id}/status | Змінити статус |
| POST | /api/feedback/{id}/vote | Проголосувати за зворотний зв'язок |
| DELETE | /api/feedback/{id}/vote | Видалити голос |
| POST | /api/feedback/{id}/comments | Додати коментар |
| GET | /api/feedback/stats | Отримати статистику зворотного зв'язку (за типом, за статусом) |

## Бізнес-правила

- Один голос на email на елемент зворотного зв'язку
- VoteCount є денормалізованим (оновлюється при голосуванні/відміні голосу)
- Не можна голосувати за власний зворотний зв'язок
- Переходи статусу повинні відповідати робочому процесу (Open -> UnderReview -> Planned -> InProgress -> Done)
- Офіційні коментарі позначаються окремо

## Обсяг тестування

- **Модульні тести**: Валідація переходів статусу, логіка підрахунку голосів, запобігання голосуванню за себе
- **Інтеграційні тести (WebApplicationFactory)**: Життєвий цикл зворотного зв'язку, голосування, коментування, робочий процес статусів
- **Тести бази даних (Testcontainers)**: Унікальність голосу, консистентність кількості голосів, аудит переходів статусу
- **Тести продуктивності (k6)**: Навантажувальне тестування списку зворотного зв'язку з сортуванням, стрес-тестування одночасного голосування

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Feedback**: Title, Description, AuthorName, AuthorEmail, CreatedAt
- **Vote**: VoterEmail, CreatedAt
- **Comment**: AuthorName, Content, CreatedAt

> Поля Type, Status, Priority, VoteCount, FeedbackId, IsOfficial мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних повинна бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Приклад розподілу повинен відображати реальні співвідношення між батьківськими та дочірніми сутностями.
