# Завдання 7: Бронювання готелю

## Предметна область

Система бронювання готельних номерів з типами номерів та управлінням гостями.

## Сутності

- **Room**: Id, Number, Type (Single/Double/Suite), PricePerNight, Floor, IsAvailable
- **Guest**: Id, FirstName, LastName, Email, Phone, PassportNumber
- **Reservation**: Id, RoomId, GuestId, CheckInDate, CheckOutDate, TotalPrice, Status (Confirmed/CheckedIn/CheckedOut/Cancelled)

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/rooms | Отримати всі номери (фільтр за типом, доступністю, діапазоном дат) |
| GET | /api/rooms/available?checkIn={date}&checkOut={date} | Отримати доступні номери на вказані дати |
| POST | /api/reservations | Створити бронювання |
| GET | /api/reservations/{id} | Отримати деталі бронювання |
| PATCH | /api/reservations/{id}/checkin | Заселити гостя |
| PATCH | /api/reservations/{id}/checkout | Виселити гостя |
| DELETE | /api/reservations/{id} | Скасувати бронювання |

## Бізнес-правила

- Номер не може бути заброньований двічі на дати, що перетинаються
- Дата виїзду має бути після дати заїзду
- Загальна вартість = кількість ночей × ціна за ніч
- Заселення дозволено лише в дату заїзду
- Скасування неможливе після заселення

## Обсяг тестування

- **Модульні тести**: Виявлення перетину дат, обчислення ціни, правила заселення/виселення
- **Інтеграційні тести (WebApplicationFactory)**: Процес бронювання, ендпоінт доступності, запобігання подвійному бронюванню
- **Тести бази даних (Testcontainers)**: Обмеження перетину бронювань, історія бронювань гостя, відстеження статусу номера
- **Тести продуктивності (k6)**: Навантажувальний тест пошуку доступності, стрес-тест конкурентних спроб бронювання

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Room**: Number, Floor
- **Guest**: FirstName, LastName, Email, Phone, PassportNumber
- **Reservation**: CreatedAt

> Поля Type, PricePerNight, IsAvailable, CheckInDate, CheckOutDate, TotalPrice, Status мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних має бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Розподіл має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
