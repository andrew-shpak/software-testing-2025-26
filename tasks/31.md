# Завдання 31: Планувальник записів

## Предметна область

Бронювання записів на послуги з постачальниками, послугами та часовими слотами.

## Сутності

- **Provider**: Id, Name, Specialization, Email, WorkingHours (початок/кінець)
- **Service**: Id, Name, DurationMinutes, Price, Description
- **Appointment**: Id, ProviderId, ServiceId, ClientName, ClientEmail, Date, StartTime, EndTime, Status (Booked/Completed/Cancelled/NoShow)

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/providers | Отримати всіх постачальників |
| POST | /api/providers | Зареєструвати постачальника |
| GET | /api/services | Отримати всі послуги |
| POST | /api/services | Створити послугу |
| GET | /api/providers/{id}/slots?date={date} | Отримати доступні часові слоти |
| POST | /api/appointments | Забронювати запис |
| PATCH | /api/appointments/{id}/cancel | Скасувати запис |
| PATCH | /api/appointments/{id}/complete | Позначити як завершений |
| GET | /api/providers/{id}/appointments?date={date} | Отримати розклад постачальника |

## Бізнес-правила

- Записи не можуть перетинатися для одного постачальника
- EndTime = StartTime + Service.DurationMinutes
- Записи мають бути в межах робочих годин постачальника
- Скасування дозволене лише за 2+ години до початку
- Доступні слоти обчислюються шляхом віднімання зайнятих слотів від робочих годин

## Обсяг тестування

- **Модульні тести**: Обчислення доступності слотів, виявлення перетинів, вікно скасування
- **Інтеграційні тести (WebApplicationFactory)**: Потік бронювання, доступність слотів, правила скасування
- **Тести бази даних (Testcontainers)**: Запобігання перетину записів, запити розкладу, зв'язки постачальник-послуга
- **Тести продуктивності (k6)**: Навантажувальне тестування доступності слотів, стрес-тестування одночасного бронювання одного слота

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Provider**: Name, Email
- **Service**: Name, Description
- **Appointment**: ClientName, ClientEmail

> Поля Specialization, WorkingHours, DurationMinutes, Price, ProviderId, ServiceId, Date, StartTime, EndTime, Status мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних має бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Приклад розподілу має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
