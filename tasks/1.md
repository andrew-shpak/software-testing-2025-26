# Task 1: Book Store

## Domain

Online book store managing books, authors, and categories.

## Entities

- **Book**: Id, Title, ISBN, Price, PublishedDate, AuthorId, CategoryId
- **Author**: Id, Name, Bio, Country
- **Category**: Id, Name, Description

## Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| GET | /api/books | Get all books (with pagination, filtering by category/author) |
| GET | /api/books/{id} | Get book by ID |
| POST | /api/books | Create a new book |
| PUT | /api/books/{id} | Update a book |
| DELETE | /api/books/{id} | Delete a book |
| GET | /api/authors | Get all authors |
| POST | /api/authors | Create an author |
| GET | /api/books/search?query={query} | Search books by title or ISBN |

## Business Rules

- ISBN must be unique and valid (13 digits)
- Price must be positive
- Book cannot be created without an existing author
- Deleting an author with associated books should be prevented

## Testing Scope

- **Unit Tests**: Book validation (ISBN format, price range), search logic, pagination
- **Integration Tests (WebApplicationFactory)**: Full CRUD endpoint testing, query parameter filtering, error responses (404, 400, 409)
- **Database Tests (Testcontainers)**: Unique ISBN constraint, cascade behavior, author-book relationship integrity
- **Performance Tests (k6)**: Load test GET /api/books with pagination, stress test search endpoint

## AutoFixture Fields

Fields to auto-generate with AutoFixture (not business-critical, just need valid data):

- **Book**: Title, Description, PublishedDate
- **Author**: Name, Bio, Country
- **Category**: Name, Description

> Fields like ISBN, Price, AuthorId, CategoryId have business rules and should be set explicitly in tests.

## Database

Use **PostgreSQL** as the database engine. For Testcontainers, use `Testcontainers.PostgreSql` package.

## Database Seeding

For performance and integration tests, the database must be pre-seeded with **at least 10,000 records** distributed across all entities. Use AutoFixture (or Bogus) to generate realistic test data. Example distribution should reflect real-world ratios between parent and child entities.
