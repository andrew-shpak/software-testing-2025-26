# Task 18: Restaurant Menu

## Domain

Restaurant menu management with dishes, categories, and daily specials.

## Entities

- **Category**: Id, Name, Description, SortOrder
- **Dish**: Id, Name, Description, Price, CategoryId, IsAvailable, Calories, Allergens (list)
- **DailySpecial**: Id, DishId, Date, DiscountPercent, SpecialPrice

## Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| GET | /api/menu | Get full menu grouped by category |
| GET | /api/categories | Get all categories |
| POST | /api/categories | Create a category |
| GET | /api/dishes | Get dishes (filter by category, available only, allergen-free) |
| POST | /api/dishes | Create a dish |
| PUT | /api/dishes/{id} | Update a dish |
| PATCH | /api/dishes/{id}/availability | Toggle dish availability |
| POST | /api/specials | Create a daily special |
| GET | /api/specials/today | Get today's specials |

## Business Rules

- Price must be positive
- DiscountPercent must be 1-50%
- SpecialPrice = Price × (1 - DiscountPercent/100)
- Only one special per dish per day
- Category SortOrder determines menu display order

## Testing Scope

- **Unit Tests**: Special price calculation, discount validation, allergen filtering
- **Integration Tests (WebApplicationFactory)**: Menu retrieval, special creation, availability toggle
- **Database Tests (Testcontainers)**: Unique special per dish per day, category ordering, dish-category relationship
- **Performance Tests (k6)**: Load test full menu retrieval, stress test daily specials endpoint

## AutoFixture Fields

Fields to auto-generate with AutoFixture (not business-critical, just need valid data):

- **Category**: Name, Description
- **Dish**: Name, Description, Allergens
- **DailySpecial**: Date

> Fields like Price, SortOrder, IsAvailable, Calories, CategoryId, DiscountPercent, SpecialPrice have business rules and should be set explicitly in tests.

## Database

Use **PostgreSQL** as the database engine. For Testcontainers, use `Testcontainers.PostgreSql` package.

## Database Seeding

For performance and integration tests, the database must be pre-seeded with **at least 10,000 records** distributed across all entities. Use AutoFixture (or Bogus) to generate realistic test data. Example distribution should reflect real-world ratios between parent and child entities.
