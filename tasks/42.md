# Завдання 42: Аптечний магазин

## Предметна область

Система управління аптечним інвентарем та рецептами.

## Сутності

- **Medicine**: Id, Name, GenericName, Manufacturer, Price, StockQuantity, ExpiryDate, RequiresPrescription, Category (Antibiotic/Painkiller/Vitamin/Cardiac/Other)
- **Prescription**: Id, PatientName, PatientPhone, DoctorName, DoctorLicense, IssuedDate, ExpiresDate, Status (Active/Fulfilled/Expired)
- **PrescriptionItem**: Id, PrescriptionId, MedicineId, Dosage, Quantity, Instructions
- **Sale**: Id, PrescriptionId (nullable), Items, TotalAmount, SoldAt

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/medicines | Отримати ліки (фільтр за категорією, вимогою рецепта, наявністю) |
| POST | /api/medicines | Додати ліки |
| PUT | /api/medicines/{id} | Оновити інформацію про ліки |
| GET | /api/medicines/expiring?days={n} | Отримати ліки, що закінчуються протягом N днів |
| GET | /api/medicines/low-stock | Отримати ліки з низьким запасом |
| POST | /api/prescriptions | Створити рецепт |
| GET | /api/prescriptions/{id} | Отримати деталі рецепта |
| POST | /api/sales | Обробити продаж |
| GET | /api/sales | Отримати історію продажів |

## Бізнес-правила

- Рецептурні ліки не можна продавати без дійсного рецепта
- Рецепт закінчується через IssuedDate + 30 днів
- Не можна продавати прострочені ліки
- Запас зменшується при продажу
- Продаж з рецептом позначає його як виконаний
- DoctorLicense повинен бути верифікований (непорожній, валідний формат)
- Поріг низького запасу: 10 одиниць

## Обсяг тестування

- **Модульні тести**: Перевірка закінчення терміну рецепта, валідація запасу, перевірка вимоги рецепта
- **Інтеграційні тести (WebApplicationFactory)**: Продаж з/без рецепта, фільтрація за терміном дії, сповіщення про низький запас
- **Тести бази даних (Testcontainers)**: Консистентність запасу при одночасних продажах, зв'язок рецепт-продаж, запити за терміном придатності
- **Тести продуктивності (k6)**: Навантажувальне тестування пошуку ліків, стрес-тестування одночасних продажів (година пік в аптеці)

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Medicine**: Name, GenericName, Manufacturer
- **Prescription**: PatientName, PatientPhone, DoctorName
- **PrescriptionItem**: Dosage, Instructions
- **Sale**: SoldAt

> Поля Price, StockQuantity, ExpiryDate, RequiresPrescription, Category, DoctorLicense, IssuedDate, ExpiresDate, Status, Quantity, TotalAmount, PrescriptionId мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних повинна бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Приклад розподілу повинен відображати реальні співвідношення між батьківськими та дочірніми сутностями.
