# Task 29: Sports League

## Domain

Sports league management with teams, players, matches, and standings.

## Entities

- **Team**: Id, Name, City, Founded, LogoUrl, CoachName
- **Player**: Id, TeamId, FirstName, LastName, Position, JerseyNumber, DateOfBirth
- **Match**: Id, HomeTeamId, AwayTeamId, MatchDate, HomeScore, AwayScore, Status (Scheduled/InProgress/Completed), Venue

## Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| GET | /api/teams | Get all teams |
| POST | /api/teams | Create a team |
| GET | /api/teams/{id}/players | Get team roster |
| POST | /api/players | Register a player |
| POST | /api/matches | Schedule a match |
| PATCH | /api/matches/{id}/score | Update match score |
| PATCH | /api/matches/{id}/complete | Mark match as completed |
| GET | /api/standings | Get league standings (W/L/D, points, goal difference) |
| GET | /api/matches?team={id} | Get matches for a team |

## Business Rules

- JerseyNumber must be unique per team
- A team cannot play against itself
- Two matches cannot overlap for the same team on the same day
- Standings: Win = 3 pts, Draw = 1 pt, Loss = 0 pts
- Match score can only be updated while InProgress

## Testing Scope

- **Unit Tests**: Standings calculation, jersey uniqueness, scheduling conflict detection
- **Integration Tests (WebApplicationFactory)**: Match lifecycle, score updates, standings endpoint
- **Database Tests (Testcontainers)**: Jersey uniqueness constraint, standings aggregation query, match scheduling conflicts
- **Performance Tests (k6)**: Load test standings calculation, stress test concurrent score updates

Use **PostgreSQL** as the database engine. For Testcontainers, use `Testcontainers.PostgreSql` package.

## Database Seeding

For performance and integration tests, the database must be pre-seeded with **at least 10,000 records** distributed across all entities. Use AutoFixture (or Bogus) to generate realistic test data. Example distribution should reflect real-world ratios between parent and child entities.
