# Завдання 29: Спортивна ліга

## Предметна область

Управління спортивною лігою з командами, гравцями, матчами та турнірною таблицею.

## Сутності

- **Team**: Id, Name, City, Founded, LogoUrl, CoachName
- **Player**: Id, TeamId, FirstName, LastName, Position, JerseyNumber, DateOfBirth
- **Match**: Id, HomeTeamId, AwayTeamId, MatchDate, HomeScore, AwayScore, Status (Scheduled/InProgress/Completed), Venue

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/teams | Отримати всі команди |
| POST | /api/teams | Створити команду |
| GET | /api/teams/{id}/players | Отримати склад команди |
| POST | /api/players | Зареєструвати гравця |
| POST | /api/matches | Запланувати матч |
| PATCH | /api/matches/{id}/score | Оновити рахунок матчу |
| PATCH | /api/matches/{id}/complete | Позначити матч як завершений |
| GET | /api/standings | Отримати турнірну таблицю (П/П/Н, очки, різниця голів) |
| GET | /api/matches?team={id} | Отримати матчі команди |

## Бізнес-правила

- JerseyNumber має бути унікальним у межах команди
- Команда не може грати проти себе
- Два матчі не можуть збігатися для однієї команди в один день
- Таблиця: Перемога = 3 очки, Нічия = 1 очко, Поразка = 0 очок
- Рахунок матчу можна оновлювати лише під час статусу InProgress

## Обсяг тестування

- **Модульні тести**: Обчислення турнірної таблиці, унікальність номерів, виявлення конфліктів розкладу
- **Інтеграційні тести (WebApplicationFactory)**: Життєвий цикл матчу, оновлення рахунку, ендпоінт турнірної таблиці
- **Тести бази даних (Testcontainers)**: Обмеження унікальності номера, запит агрегації турнірної таблиці, конфлікти розкладу матчів
- **Тести продуктивності (k6)**: Навантажувальне тестування обчислення турнірної таблиці, стрес-тестування одночасних оновлень рахунку

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Team**: Name, City, Founded, LogoUrl, CoachName
- **Player**: FirstName, LastName
- **Match**: Venue

> Поля Position, JerseyNumber, DateOfBirth, TeamId, HomeTeamId, AwayTeamId, MatchDate, HomeScore, AwayScore, Status мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних має бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Приклад розподілу має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
