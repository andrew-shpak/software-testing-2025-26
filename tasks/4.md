# Завдання 4: Трекер завдань

## Предметна область

Система управління проєктними завданнями з дошками та призначенням завдань.

## Сутності

- **Board**: Id, Name, Description, CreatedAt
- **TaskItem**: Id, BoardId, Title, Description, Status (Todo/InProgress/Done), Priority (Low/Medium/High/Critical), AssigneeId, DueDate, CreatedAt
- **Assignee**: Id, Name, Email

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/boards | Отримати всі дошки |
| POST | /api/boards | Створити дошку |
| GET | /api/boards/{id}/tasks | Отримати завдання дошки (фільтр за статусом, пріоритетом) |
| POST | /api/boards/{id}/tasks | Створити завдання на дошці |
| PUT | /api/tasks/{id} | Оновити завдання |
| PATCH | /api/tasks/{id}/status | Змінити статус завдання |
| DELETE | /api/tasks/{id} | Видалити завдання |
| GET | /api/tasks/overdue | Отримати всі прострочені завдання |

## Бізнес-правила

- Переходи статусів: Todo → InProgress → Done (без пропуску назад)
- Термін виконання має бути в майбутньому при створенні завдання
- Завдання з критичним пріоритетом повинні мати виконавця
- Назва дошки має бути унікальною

## Обсяг тестування

- **Модульні тести**: Валідація переходів статусів, правило пріоритет-виконавець, логіка визначення прострочених завдань
- **Інтеграційні тести (WebApplicationFactory)**: CRUD робочі процеси, ендпоінт зміни статусу, фільтрація, ендпоінт прострочених завдань
- **Тести бази даних (Testcontainers)**: Обмеження унікальності назви дошки, зв'язок завдання-дошка, масове оновлення статусів
- **Тести продуктивності (k6)**: Навантажувальний тест списку завдань дошки з фільтрами, стрес-тест конкурентного створення завдань

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Board**: Name, Description, CreatedAt
- **TaskItem**: Title, Description, CreatedAt
- **Assignee**: Name, Email

> Поля Status, Priority, DueDate, AssigneeId, BoardId мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних має бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Розподіл має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
