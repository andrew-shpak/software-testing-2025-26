# Завдання 27: Сервіс підписок

## Предметна область

Управління SaaS-підписками з тарифними планами, користувачами та платіжними циклами.

## Сутності

- **Plan**: Id, Name, Description, MonthlyPrice, Features (список), MaxUsers, IsActive
- **Subscription**: Id, UserId, PlanId, StartDate, EndDate, Status (Active/Cancelled/Expired), AutoRenew
- **Invoice**: Id, SubscriptionId, Amount, IssuedAt, DueDate, Status (Pending/Paid/Overdue)

## Ендпоінти

| Метод | Маршрут | Опис |
|--------|-------|-------------|
| GET | /api/plans | Отримати доступні плани |
| POST | /api/plans | Створити план |
| POST | /api/subscriptions | Підписатися на план |
| GET | /api/subscriptions/{id} | Отримати деталі підписки |
| PATCH | /api/subscriptions/{id}/cancel | Скасувати підписку |
| PATCH | /api/subscriptions/{id}/upgrade | Змінити на інший план |
| GET | /api/subscriptions/{id}/invoices | Отримати рахунки за підпискою |
| POST | /api/invoices/{id}/pay | Оплатити рахунок |

## Бізнес-правила

- Користувач може мати лише одну активну підписку
- Скасування набирає чинності наприкінці платіжного періоду
- Оновлення плану набирає чинності негайно з пропорційною оплатою
- Рахунки генеруються автоматично щомісяця
- Прострочення після 30 днів від дати оплати

## Обсяг тестування

- **Модульні тести**: Обчислення пропорційної оплати, логіка платіжного циклу, виявлення прострочення
- **Інтеграційні тести (WebApplicationFactory)**: Потік підписки/скасування/оновлення, генерація рахунків, обробка платежів
- **Тести бази даних (Testcontainers)**: Обмеження на єдину активну підписку, цілісність ланцюжка рахунків, історія зміни планів
- **Тести продуктивності (k6)**: Навантажувальне тестування списку планів, стрес-тестування одночасних змін підписок

## Поля AutoFixture

Поля для автоматичної генерації за допомогою AutoFixture (не критичні для бізнес-логіки, потрібні лише валідні дані):

- **Plan**: Name, Description, Features
- **Subscription**: CreatedAt
- **Invoice**: IssuedAt

> Поля MonthlyPrice, MaxUsers, IsActive, StartDate, EndDate, Status, AutoRenew, Amount, DueDate мають бізнес-правила і повинні встановлюватися явно в тестах.

## База даних

Використовуйте **PostgreSQL** як двигун бази даних. Для Testcontainers використовуйте пакет `Testcontainers.PostgreSql`.

## Наповнення бази даних

Для тестів продуктивності та інтеграційних тестів база даних має бути попередньо наповнена **щонайменше 10 000 записами**, розподіленими між усіма сутностями. Використовуйте AutoFixture (або Bogus) для генерації реалістичних тестових даних. Приклад розподілу має відображати реальні співвідношення між батьківськими та дочірніми сутностями.
