# Task 27: Subscription Service

## Domain

SaaS subscription management with plans, users, and billing cycles.

## Entities

- **Plan**: Id, Name, Description, MonthlyPrice, Features (list), MaxUsers, IsActive
- **Subscription**: Id, UserId, PlanId, StartDate, EndDate, Status (Active/Cancelled/Expired), AutoRenew
- **Invoice**: Id, SubscriptionId, Amount, IssuedAt, DueDate, Status (Pending/Paid/Overdue)

## Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| GET | /api/plans | Get available plans |
| POST | /api/plans | Create a plan |
| POST | /api/subscriptions | Subscribe to a plan |
| GET | /api/subscriptions/{id} | Get subscription details |
| PATCH | /api/subscriptions/{id}/cancel | Cancel a subscription |
| PATCH | /api/subscriptions/{id}/upgrade | Change to a different plan |
| GET | /api/subscriptions/{id}/invoices | Get invoices for subscription |
| POST | /api/invoices/{id}/pay | Pay an invoice |

## Business Rules

- User can have only one active subscription
- Cancellation takes effect at end of billing period
- Upgrade takes effect immediately with prorated charge
- Invoices auto-generated monthly
- Overdue after 30 days past due date

## Testing Scope

- **Unit Tests**: Proration calculation, billing cycle logic, overdue detection
- **Integration Tests (WebApplicationFactory)**: Subscribe/cancel/upgrade flow, invoice generation, payment processing
- **Database Tests (Testcontainers)**: Single active subscription constraint, invoice chain integrity, plan change history
- **Performance Tests (k6)**: Load test plan listing, stress test concurrent subscription changes

## AutoFixture Fields

Fields to auto-generate with AutoFixture (not business-critical, just need valid data):

- **Plan**: Name, Description, Features
- **Subscription**: CreatedAt
- **Invoice**: IssuedAt

> Fields like MonthlyPrice, MaxUsers, IsActive, StartDate, EndDate, Status, AutoRenew, Amount, DueDate have business rules and should be set explicitly in tests.

## Database

Use **PostgreSQL** as the database engine. For Testcontainers, use `Testcontainers.PostgreSql` package.

## Database Seeding

For performance and integration tests, the database must be pre-seeded with **at least 10,000 records** distributed across all entities. Use AutoFixture (or Bogus) to generate realistic test data. Example distribution should reflect real-world ratios between parent and child entities.
